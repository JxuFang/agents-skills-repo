# 第一性原理分析技能

assistant 处理复杂决策和开放式问题的执行规范。先拆清楚问题结构，再从底层重建方案，给出可验证的下一步。

---

## 触发与退出

**启用条件**（满足任一）：
- 用户明确要求第一性原理分析
- 问题存在多个可行解，需要做取舍
- 讨论围绕方案展开，但目标和约束不清楚
- 现有方案建立在默认假设、行业惯例或路径依赖上

触发信号词："从本质上看" / "回到目标" / "不要套模板" / "重新想这件事" / "为什么要这么做"

**不启用**：任务边界明确只需执行 / 查询润色整理 / 线上止血 / 用户要的是 best practice / 关键事实严重缺失应先补信息。

**如果问题不需要重建，就不要为了显得深刻而重建。**

---

## 与用户交互

**先追问再分析的情况**：
1. 目标不明（用户只给了手段没给目标）→ 追问真正想达成的结果
2. 关键约束缺失（无法判断方案可行性）→ 追问预算、时间、团队等
3. 关键事实缺失（结论会因此翻转）→ 指出并追问

**追问方式**：说明为什么需要这个信息；给 2-3 个选项而非开放式提问；一次不超过 3 个问题。

**不追问**：可从上下文合理推断的信息，标注为假设即可；用户说"你来判断"或"先分析看看"。

---

## 工作流

根据复杂度选择深度：目标清晰约束明确 → 精简版（省略 Step 2-5，直接给建议）；目标需确认、约束部分隐含 → 标准版（可合并步骤）；目标模糊、约束复杂、多方利益 → 完整版。

### Step 1：重述问题，分离目标与手段

用一句话重述：**为了【目标】，在【约束】下，需要决定【问题】。**

用户说的方案 ≠ 目标。"要不要上微服务"是手段，"如何提升发布效率和故障隔离"才是目标。如果原问题混淆了目标和方案，先拆开。如果目标不明，先追问。

### Step 2：明确目标与成功标准

- 核心目标是什么（结果导向，非动作导向）
- 什么结果算成功（尽量量化或可观测）
- 什么不是这次的目标（防止范围蔓延）

### Step 3：列出约束

- **硬约束**：违反则不可行（预算、法规、时间、安全、兼容性、团队规模）
- **软约束**：违反则不理想但仍可行（偏好、习惯、熟悉度、风格一致性）

有些约束只是被默认接受，实际上可以协商或绕过——遇到时主动指出。

### Step 4：整理事实、未知与假设

分三类列清楚：

| 已知事实 | 未知信息 | 关键假设 |
|---------|---------|---------|
| 有依据，标明来源 | 缺失但影响结论 | 正在影响决策但未验证 |

对每个关键假设必须补充：
- 如果错了，结论会怎样翻转
- 如何验证或证伪（实验、数据对比、试点等）

禁止模糊假设。不合格："微服务应该更好"。合格："如果团队有 3+ 模块的并行发布需求，微服务能将发布冲突降到接近 0，可通过统计当前冲突频率验证"。

### Step 5：拆到底层机制

不要停在"微服务""自动化""平台化"等标签层面。持续追问：

- 当前痛点**为什么**会出现？因果链是什么？
- 哪些前提只是惯例，不是物理/逻辑必需？
- 如果从零开始设计，会得到什么不同方案？

直到触达基本因子：用户需求、信息流瓶颈、错误来源、成本结构、协作瓶颈、性能极限、风险传播路径、反馈循环。

### Step 6：重建方案

基于底层机制给出 2-3 条路径（不为凑数而虚构）：

| 维度 | 保守路径 | 平衡路径 | 激进路径 |
|------|---------|---------|---------|
| 定位 | 改动小、验证快、可逆 | 收益与代价均衡 | 潜在收益大、风险高 |
| 适用条件 | | | |
| 预期收益 | | | |
| 主要代价 | | | |
| 关键风险 | | | |
| 优先验证点 | | | |

主动检测思维陷阱：
- 只列了支持证据 → 强制补反对证据
- "大厂都这么做" → 追问他们的条件和我们一样吗
- "要么 A 要么 B" → 是否存在 C？能否组合？
- 不敢否定已有投入 → 如果从零开始，还会选现状吗？

### Step 7：给出建议与下一步

- 推荐哪条路径，为什么
- 下一步先做什么（具体动作，不是"进一步研究"）
- 如果信息不足，先补什么事实、怎么补

**推荐优先级**：验证成本低 > 可逆性高 > 与当前约束匹配。

## 输出前检查清单

每次输出前逐条过：

1. 目标和手段分开了吗？没有目标不进入方案比较
2. 约束列了吗？没有约束不讨论最优
3. 事实和假设分开了吗？无证据的必须标记为假设
4. 每个假设给了验证方法吗？
5. 每个方案都写了代价和风险吗？只有优点 = 分析不到位
6. 建议落到了具体下一步吗？"进一步研究"不算
7. 只列了支持证据？强制补反对证据
8. 用了抽象标签代替因果分析吗？"需要平台化"不是分析
9. 这个问题真的需要第一性原理分析吗？

## 示例

**用户**："我们团队 6 个人，Django 单体越来越难维护，是不是应该拆微服务？"

**分析骨架**（标准版）：

- **问题重述**：为了提高可维护性和开发效率，在 6 人团队 Django 技术栈约束下，决定如何改善单体维护困难。（"微服务"是手段不是目标）
- **关键假设**："难维护"源于模块耦合而非代码质量（验证：统计 3 个月 bug 和 merge conflict 归因）；团队有能力运维分布式系统（验证：问团队容器编排经验）
- **底层因果**：模块边界不清 → 改 A 动 B → conflict 多 → 上线慢。问题可能是代码组织而非架构问题
- **方案**：A 单体内模块化重构（2 周验证）→ B 渐进拆最痛模块（1 月验证）→ C 全面微服务（6 人运维 4-6 服务负担极重）
- **建议**：先 A，验证模块化是否够用，再决定是否进 B

**第一性原理不是拒绝经验，而是不被经验绑架。**
